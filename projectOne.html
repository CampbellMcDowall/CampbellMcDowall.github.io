Project One

For this Project I was on the Internet of Things team, we were tasked to create a LoRaWAN network that was to be made freely available for public use.


The plan

This semister we were aiming to have a working LoRaWAN network in Dunedin, New Zealand. Meaning at least one gateway that connects to a backend server such as the Internet Of Things or Lora server.

At first we were looking at getting up to 5 gateways; three of which would cover parts of dunedin, one would be for a future deployment and the last would be for in house development. We soon realised that it was not going to be cheap to implement a solution that would work and cover the range we required. We looked at the previous teams work and decided that we would need to start from scratch due to the many problems they were having with hardware incompatibilitys etc.


My Contributions

For the first few weeks The team worked together to look through the previous teams work and see if we could salvage any of it, we soon decided that it was going to save more time if we re-evaluated the whole project and started from the beginning. We spent a solid amount of time researching new gateways that had became available and also solutions that also where not available options for the previous team.

I assisted in contacting various companies for information and prices of their gateways, finding a suitable gateway at a reasonable price was difficult but many companies where interested in our project. The main problem we had with choosing a gateway was finding one that we could use legally in New Zealand, after research and talking to some interested parties we knew we had to be ideally between 915 and 925mhz.
Finding a gateway that allowed for these frequencies was difficult



From my research I definitely knew that it was going to be difficult to get one of the gateways and nodes working if we went with a cheap or Do it yourself option. If the budget would have allowed it would have been ideal to use a solution by Kerlink or Libelium as these companies offer proper support and proven working systems. Implementation of the gateways would potentially involve waterproofing, lightning protection, installation of the gateway on top of a building or other high structure. For the installation it was likely that a professional would have to install it and permission would need to be granted.

The first gateway we purchased from Haxiot and unfortunatly it did not meet our expectations or as expected, code that was provided did not work properly and required the team to look at and reconfigure the code to run. It was not a good start tothe project and dampened spirits slightly. Now we have a Multitech Conduit which I think is a good solution by Multitech that will work great for our situation, it wasnt the most expensive option which will allow us to get more gateways when required. The gateway has a setting for New Zealand frequencies, it is more of an industrial solution than the Haxiot gateway and Multitech offer great support.


Tasks


Room sensor

Brendan and myself were given a task to make a room sensor which we would build based off a schematic given to us, we researched the parts and purchased them or best alternative. As the project specification changed we had to adapt the sensor and project. Sensors used in the most current version include temperature, humidity, light (lux), sound and motion. The data is collected at intervals then stored in a csv file on the micro sd card. We divided up the sensors then set off to build smaller example circuits for each or we used the available online documentation.
My main task was to join the various sensors together into one project (the final version). Originally we used an Arduino Uno which I quickly realised was not going to be big enough. The arduino Uno ran out of memory very quickly after we added the micro sd card breakout board and lux sensor. Though we tried to make the project take less space and after half a week we swapped the Uno’s for Mega’s. With more memory there was no issue writing to the micro SD card file.

Use case essay

One of our tasks was to choose a LoRaWAN use case and produce an essay about the subject. I chose Amsterdam and their use of The Things Network LoRaWAN, focusing on current uses and also concerns involved with the technology. Amsterdam is where The Things Network started and is mostly developed from, they also currently have some interesting and unique use cases. While researching for this essay it was not very easy to find reliable information on the LoRaWAN networks, many articles and eduational journals are copyied and tweaked versions of eachother. This ment there are a lot of lnks but not with a lot of useful information. Due to the LoRaWAN protocol being relatively young it is always changing and evolving with how people use it, teh standard is not fully defined so it has allowed some manufacturers to put their own take on how it should be best implemented. This is good for innovation but can be difficult when you want solid information about setting up the best possible network. To view my essay on Amsterdam and The Things Network basic TTN install guide

After the team having mixed results with the Haxiot gateway, I decided to do some further research an to reach out to Haxiot. From this I learnt the following "The Gateway to the NS uses SSL to connect. You will need port 443/TCP open from your virtual Gateway to the Network Server. After checking with the Technician Rob I found that this shouldn’t be any problem as the ports are open.
